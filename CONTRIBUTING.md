## 🤔 How Can I Contribute?

### 🐛 Reporting Bugs

If you find a bug, please [open an issue](https://github.com/kambodia1/instagram-login-new/issues/new) with the following information:

- **Description** — A clear and concise description of the bug
- **Steps to Reproduce** — Detailed steps to reproduce the behavior
- **Expected Behavior** — What you expected to happen
- **Screenshots** — If applicable, add screenshots to help explain the issue
- **Environment** — Browser, OS, and device information

### 💡 Suggesting Features

Feature requests are welcome! Please [open an issue](https://github.com/kambodia1/instagram-login-new/issues/new) and include:

- A clear and descriptive title
- A detailed description of the proposed feature
- Any relevant mockups or examples
- Why this feature would be useful

### 📝 Improving Documentation

Documentation improvements are always welcome. This includes:

- Fixing typos or grammatical errors
- Adding examples or clarifications
- Updating outdated information

### 💻 Submitting Code Changes

Want to contribute code? Awesome! Please follow the steps below.

## 🚀 Getting Started

**1. Fork the repository**

Click the **Fork** button on the top right of the [repository page](https://github.com/kambodia1/instagram-login-new).

**2. Clone your fork**

```bash
git clone https://github.com/your-username/instagram-login-new.git
cd instagram-login-new
```

**3. Create a new branch**

```bash
git checkout -b feature/your-feature-name
```

**4. Install dependencies**

```bash
bun install
# or
npm install
```

**5. Make your changes**

Edit files, add features, fix bugs — do your thing!

**6. Test your changes locally**

```bash
bun run dev
# or
npm run dev
```

Open `http://localhost:8888` and verify everything works correctly.

## 🔄 Pull Request Process

1. **Ensure your code works** — Test thoroughly before submitting
2. **Update documentation** — If your changes affect the README or other docs, update them accordingly
3. **Follow style guidelines** — See below
4. **Write a descriptive PR title and description** — Explain *what* you changed and *why*
5. **Reference related issues** — Link any related issues (e.g., `Closes #12`)
6. **Request a review** — Tag the maintainer for review

### PR Template

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Refactoring
- [ ] Other (please describe)

## Checklist
- [ ] I have tested my changes locally
- [ ] I have updated the documentation (if applicable)
- [ ] My code follows the project's style guidelines
- [ ] I have read the CONTRIBUTING.md
```

## 🎨 Style Guidelines

### HTML
- Use semantic HTML5 elements
- Keep proper indentation (4 spaces)
- Add descriptive comments in **English** for code sections
- Keep visible text content in its original language

### CSS
- Use meaningful class names
- Group related properties together
- Add comments for non-obvious styling decisions

### JavaScript
- Use `const` and `let` — never `var`
- Use arrow functions where appropriate
- Add comments in **English** explaining complex logic
- Use camelCase for variable and function names

### General
- All code comments must be in **English**
- Visible UI text (labels, messages, etc.) should remain in their original language
- Keep files clean and well-organized

## 📝 Commit Convention

We follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
<type>(<scope>): <description>
```

### Types

| Type | Description |
|------|-------------|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation changes |
| `style` | Code style changes (formatting, no logic change) |
| `refactor` | Code refactoring |
| `test` | Adding or updating tests |
| `chore` | Maintenance tasks |

### Examples

```bash
git commit -m "feat: add dark mode toggle"
git commit -m "fix: resolve modal not closing on overlay click"
git commit -m "docs: update installation instructions"
git commit -m "style: format CSS with consistent indentation"
```
